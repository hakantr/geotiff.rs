A native asynchronous Rust port of
[geotiff.js 3.1.0](https://github.com/geotiffjs/geotiff.js/tree/v3.1.0) for
reading, decoding, transforming, and writing TIFF/GeoTIFF raster data without a
JavaScript or browser runtime.

The crate release is deliberately versioned exactly `3.1.0`, matching the
upstream API and behavior baseline. Live differential tests use geotiff.js
commit `8594d1b4bde4072326916185c848e73a9e704850`. The complete compatibility
contract, verified differences, and acceptance evidence are recorded in
[PORTING_PLAN.md](https://github.com/hakantr/geotiff.rs/blob/main/PORTING_PLAN.md).

The Mount Yasur SAR preview in this README was generated by opening the pinned
real GeoTIFF with this Rust port, decoding it through `read_rgb`, and resampling
it to 800 x 800 with bilinear interpolation. The PNG was reopened and verified
pixel for pixel. The status badges also belong to this repository's own
workflows. No visual asset from the geotiff.js README is reused.

![Mount Yasur SAR GeoTIFF output produced by geotiff.rs](https://raw.githubusercontent.com/hakantr/geotiff.rs/main/docs/assets/mount-yasur-sar.png)

## Features

- Open GeoTIFFs from local files, memory, HTTP/HTTPS range sources, custom
  asynchronous readers, and S3/GCS/Azure/local `object_store` implementations.
- Combine a main image with one or more external overview files.
- Read Classic TIFF and BigTIFF in either byte order without narrowing native
  64-bit integer values through `f64`.
- Read stripped or tiled images using chunky or planar configuration.
- Read unsigned/signed 1â€“64 bit integers and Float16/32/64 samples. Packed
  samples provide both lossless native and exact geotiff.js compatibility
  modes.
- Select bands and windows, fill out-of-bounds areas, return separate bands or
  pixel-interleaved output, and resize with nearest or bilinear resampling.
- Automatically choose internal or external overviews for a requested output
  resolution or bounding box.
- Convert WhiteIsZero, BlackIsZero, RGB, palette, CMYK, YCbCr, and CIELab data
  to RGB, including supported alpha samples.
- Decode uncompressed, PackBits, LZW, Deflate, JPEG, LERC, Zstandard, and WebP
  data. JPEG2000 is available as an additive native capability.
- Register or override decoders, configure compressed/decoded caches, limit
  decode concurrency, and cancel I/O/decode operations.
- Read GeoKeys, GDAL metadata/NoData, COG ghost metadata, tie points, origin,
  resolution, PixelIsArea, and bounding boxes.
- Write classic GeoTIFFs from flat typed values or nested bands, including
  multi-strip, tiled, planar, GeoKey, and GDAL metadata layouts.

## Installation

The crates.io package name `geotiff` is owned by an unrelated project, so this
port intentionally cannot publish under that registry identity. Pin the GitHub
release in `Cargo.toml`:

```toml
[dependencies]
geotiff = { git = "https://github.com/hakantr/geotiff.rs", tag = "v3.1.0" }
tokio = { version = "1", features = ["macros", "rt-multi-thread"] }
```

Native codec dependencies are built as part of the crate. A normal Rust build
toolchain plus a C/C++ compiler, CMake, and libclang must be available on the
target platform.

## Verified usage examples

These are not hypothetical API sketches. The same sources live under
`examples/`; they are compiled by the release gate and were executed against
committed fixtures or the pinned real remote file while preparing this release.

### Read a local GeoTIFF

Open the repository's real planar RGB fixture and read its first image as an
interleaved typed raster:

```rust,no_run
use geotiff::{ReadRasterResult, ReadRastersOptions, from_file};

#[tokio::main]
async fn main() -> Result<(), Box<dyn std::error::Error>> {
    let tiff = from_file("tests/fixtures/planar-rgb-u8.tif").await?;
    let image = tiff.image(0)?;

    println!("{} x {}", image.width(), image.height());
    println!("bbox: {:?}", image.bounding_box(false)?);

    let result = image
        .read_rasters(ReadRastersOptions {
            interleave: true,
            ..ReadRastersOptions::default()
        })
        .await?;

    if let ReadRasterResult::Interleaved(raster) = result {
        println!("{} samples in {:?}", raster.data.len(), raster.data);
    }
    Ok(())
}
```

Executable source:
[examples/read_file.rs](https://github.com/hakantr/geotiff.rs/blob/main/examples/read_file.rs)

```sh
cargo run --locked --example read_file
```

To open the Mount Yasur SAR GeoTIFF from its pinned corpus URL, produce the
800 x 800 preview through the port's BlackIsZero-to-RGB and bilinear resampling
paths, and verify every generated pixel after reopening it:

```sh
cargo run --locked --example render_png
```

### Read a real remote GeoTIFF

The URL pins both the commit and file in `GeoTIFF/test-data`; it is not a
placeholder endpoint:

```rust,no_run
use geotiff::{HttpSourceOptions, ReadRasterResult, ReadRastersOptions, from_url_with_options};

const URL: &str = concat!(
    "https://raw.githubusercontent.com/GeoTIFF/test-data/",
    "8506204783ff26a6c49ed1f721e7e1635b2e43ee/",
    "files/GeogToWGS84GeoKey5.tif"
);

#[tokio::main]
async fn main() -> Result<(), Box<dyn std::error::Error>> {
    let tiff = from_url_with_options(
        URL,
        HttpSourceOptions {
            allow_full_file: true,
            ..HttpSourceOptions::default()
        },
    )
    .await?;
    let image = tiff.image(0)?;
    let raster = image
        .read_rasters(ReadRastersOptions {
            interleave: true,
            ..ReadRastersOptions::default()
        })
        .await?;
    let ReadRasterResult::Interleaved(raster) = raster else {
        unreachable!("interleave=true always returns an interleaved raster");
    };

    println!("GeoKeys: {:?}", image.geo_keys());
    println!("bounding box: {:?}", image.bounding_box(false)?);
    println!("decoded values: {}", raster.data.len());
    Ok(())
}
```

Executable source:
[examples/read_remote.rs](https://github.com/hakantr/geotiff.rs/blob/main/examples/read_remote.rs)

```sh
cargo run --locked --example read_remote
```

## Windows, bands, and resampling

This example selects bands 2 and 0 from the top-left window of the committed
12-bit RGB fixture and resizes the result to 16 x 16 with bilinear resampling:

```rust,no_run
use geotiff::{ImageWindow, ReadRastersOptions, from_file};

#[tokio::main]
async fn main() -> Result<(), Box<dyn std::error::Error>> {
    let tiff = from_file("tests/fixtures/12bit.cropped.rgb.tiff").await?;
    let image = tiff.image(0)?;
    let window = ImageWindow {
        x0: 0,
        y0: 0,
        x1: image.width().min(32) as i64,
        y1: image.height().min(32) as i64,
    };
    let raster = image
        .read_rasters(ReadRastersOptions {
            window: Some(window),
            samples: vec![2, 0],
            interleave: true,
            width: Some(16),
            height: Some(16),
            resample_method: "bilinear".to_owned(),
            ..ReadRastersOptions::default()
        })
        .await?;
    assert_eq!((raster.width(), raster.height()), (16, 16));
    Ok(())
}
```

Executable source:
[examples/read_window.rs](https://github.com/hakantr/geotiff.rs/blob/main/examples/read_window.rs)

```sh
cargo run --locked --example read_window
```

`ReadRastersOptions` mirrors the observable `readRasters()` options from
geotiff.js. For packed samples, the default `PackedSampleMode::Lossless`
retains every sample. Set `PackedSampleMode::GeotiffJs` when reproducing the
exact 3.1.0 packed-offset behavior is required during migration.

## RGB conversion

The executable `read_file` example above also runs this RGB path against the
committed planar fixture:

```rust,no_run
use geotiff::{ReadRasterResult, ReadRgbOptions};

# async fn example(image: &geotiff::GeoTiffImage<'_>) -> Result<(), Box<dyn std::error::Error>> {
let rgb = image
    .read_rgb(ReadRgbOptions {
        interleave: true,
        enable_alpha: true,
        ..ReadRgbOptions::default()
    })
    .await?;

if let ReadRasterResult::Interleaved(rgb) = rgb {
    println!("RGB dimensions: {} x {}", rgb.width, rgb.height);
}
# Ok(())
# }
```

## Metadata

The remote example above executes the GeoKey and bounding-box paths. The full
image metadata API is:

```rust,no_run
# async fn example(image: &geotiff::GeoTiffImage<'_>) -> Result<(), Box<dyn std::error::Error>> {
println!("GeoKeys: {:?}", image.geo_keys());
println!("tie points: {:?}", image.tie_points()?);
println!("GDAL metadata: {:?}", image.gdal_metadata(None)?);
println!("GDAL NoData: {:?}", image.gdal_nodata());
println!("origin: {:?}", image.origin()?);
println!("resolution: {:?}", image.resolution(None)?);
println!("pixel is area: {}", image.pixel_is_area());
# Ok(())
# }
```

Numeric or named TIFF tag access is available through
`image.file_directory()`. Unlike geotiff.js's deferred arrays, the native
directory is eagerly actualized after opening.

## Writing GeoTIFFs

This self-contained example writes a 2 x 2 TIFF, reopens it through the public
reader, and verifies every raster value:

```rust,no_run
use geotiff::{
    ReadRasterResult, ReadRastersOptions, TypedArray, WriterMetadata, from_file,
    write_array_buffer,
};

#[tokio::main]
async fn main() -> Result<(), Box<dyn std::error::Error>> {
    let output = std::env::temp_dir().join("geotiff-rs-example.tif");
    let pixels = vec![1_u8, 2, 3, 4];
    let bytes = write_array_buffer(pixels.clone(), WriterMetadata::new(2, 2))?;
    std::fs::write(&output, bytes)?;

    let tiff = from_file(&output).await?;
    let raster = tiff
        .image(0)?
        .read_rasters(ReadRastersOptions {
            interleave: true,
            ..ReadRastersOptions::default()
        })
        .await?;
    let ReadRasterResult::Interleaved(raster) = raster else {
        unreachable!("interleave=true always returns an interleaved raster");
    };
    assert_eq!(raster.data, TypedArray::Uint8(pixels));
    Ok(())
}
```

Executable source:
[examples/write_geotiff.rs](https://github.com/hakantr/geotiff.rs/blob/main/examples/write_geotiff.rs)

```sh
cargo run --locked --example write_geotiff
```

The writer defaults to lossless native serialization. Use
`write_array_buffer_with_mode(..., WriterCompatibility::GeotiffJs)` only when
the exact signed-array wire quirk of the JavaScript 3.1.0 writer is required.

## JavaScript-to-Rust mapping

| geotiff.js | geotiff.rs |
|---|---|
| `fromUrl`, `fromArrayBuffer`, `fromBlob`, `fromFile` | `from_url`, `from_array_buffer`, `from_blob`, `from_file` |
| `fromCustomClient` | `from_custom_client`, `from_reader` |
| `fromUrls` | `from_urls` |
| `GeoTIFF`, `MultiGeoTIFF` | `SingleGeoTiff`, `MultiGeoTiff` |
| `GeoTIFFBase` | `GeoTiffDataset` |
| `GeoTIFFImage` | `GeoTiffImage` |
| `readRasters`, `readRGB` | `read_rasters`, `read_rgb` |
| `AbortSignal` | `CancellationToken` |
| `Pool` | native Rayon decode pool |
| `addDecoder`, `getDecoder` | `add_decoder`, `get_decoder` |
| `writeArrayBuffer`, `writeGeotiff` | `write_array_buffer`, `write_geotiff` |

Rust's explicit result, enum, and ownership types replace JavaScript's dynamic
objects while preserving valid input/output behavior. Deliberate safety or
data-preserving differences are listed in
[PORTING_PLAN.md](https://github.com/hakantr/geotiff.rs/blob/main/PORTING_PLAN.md).

## Releases and documentation

- Pushing the exact `v3.1.0` tag runs the release workflow. It verifies the tag,
  Cargo version, a clean pinned geotiff.js build, five live JS/Rust differential
  matrices, the pinned `GeoTIFF/test-data` corpus, formatting, Clippy, doctests,
  Rustdoc, and the packaged crate before creating a GitHub Release with the
  `.crate` file and SHA-256 checksum.
- Every push to `main`, and every manual dispatch, builds Rustdoc and deploys it
  to GitHub Pages. Configure the repository's Pages source as **GitHub Actions**
  before the first deployment.

No crates.io upload is attempted: the `geotiff` registry name belongs to the
unrelated `georust/geotiff` project. This guard prevents publishing the port
under another project's package identity.

## Scope and known limitation

Coordinate reprojection, a map renderer, and GPU texture management are outside
the scope of geotiff.js and this port. Native WebP RGB/RGBA decoding is tested;
the only outstanding certification item is the same WebP comparison through a
real browser `createImageBitmap` backend, which is recorded explicitly in the
porting plan.
